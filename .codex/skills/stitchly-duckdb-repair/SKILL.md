# Stitchly DuckDB Repair

Use this skill when a Stitchly workflow reports DuckDB corruption, repeated run-history mirror warnings, checksum mismatches, or adapter failures that still point at a corrupt `workflow.duckdb`.

Common signals:

- `IO Error: Corrupt database file`
- `computed checksum ... does not match stored checksum`
- repeated `failed to mirror run ... into workflow DuckDB`
- a workflow-local DB at `.stitchly/users/.../workflows/<workflow_id>/db/workflow.duckdb` needs to be recreated while preserving workflow JSON and artifacts

## Repair Workflow

1. Identify the exact workflow DB path from logs or runtime params.
   Usually it is:
   `.stitchly/users/<user>/workspaces/<workspace_id>/workflows/<workflow_id>/db/workflow.duckdb`

2. Check whether the file is open:
   `lsof <path-to-workflow.duckdb>`

3. If it is held by `stitchly-server`, stop or restart the server before repair.
   Do not rename an actively used database.

4. Run the bundled script:
   `bash .codex/skills/stitchly-duckdb-repair/scripts/repair_workflow_duckdb.sh <path-to-workflow.duckdb>`

5. Confirm the output includes a quarantine path.

6. Rerun the workflow. Stitchly recreates the DB with the embedded DuckDB library it uses at runtime. The repaired DB starts empty, so DuckDB-resident staging tables, merged tables, checkpoints, and mirrored run output need to be regenerated by the workflow. Files under `files/` and `workflow.json` are preserved.

## Notes

- The repair is intentionally a rename, not a delete. The old file is moved to `workflow.duckdb.corrupt.<timestamp>`.
- The script also quarantines `workflow.duckdb.wal` and `workflow.duckdb.tmp` if they exist.
- Do not recreate workflow DBs with an external `duckdb` CLI. A CLI-created file can be incompatible with the embedded DuckDB library linked into Stitchly and can trigger native DuckDB assertions.
- The script intentionally leaves the DB path absent after quarantine. Let Stitchly recreate it on the next run or catalog open.
