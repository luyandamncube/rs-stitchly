# 07 Dashboard Empty States

Type: `empty-state-study`

Purpose:

- define how the dashboard should feel when there is little or no operational data yet
- distinguish between different kinds of emptiness instead of using one generic blank state
- establish the right action emphasis and tone for early-stage, filtered, and all-clear moments

What this sample is testing:

- a brand-new workspace with no runs yet
- a filtered/no-results state
- a healthy “nothing to fix” state
- whether empty states can still feel premium and calm inside the contained shell

What this sample intentionally includes:

- same dashboard shell language as the other studies
- three distinct empty-state cards in one comparison surface
- action-first CTA treatment for setup
- gentler no-results and all-clear variants

What this sample intentionally does not solve yet:

- animated onboarding or guided tours
- empty states for every dashboard destination
- true product copy finalization
- mobile empty-state adaptation
- contextual illustrations or richer brand art

Shared styling:

- this sample uses `../shared.css`
- empty-state helpers are appended to the shared stylesheet rather than using a local override file

Review questions:

- do these feel like meaningfully different empty states?
- is the action emphasis too strong or too weak in the “new workspace” case?
- should the healthy/no-incidents state feel more celebratory or stay quiet?
- which of these patterns should become the default dashboard-empty language?
