SQL transform as the shape boundary between raw staging and durable merge

This study treats `sql_transform` as the place where landed raw tables stay intact while merge-ready normalized tables get materialized for downstream policy nodes. The example is the Treasury bootstrap case: wide tenor columns in, `curve_date` / `tenor` / `yield_pct` out.

Boundary

Shape changes belong here. `load_to_duckdb` should stay focused on reliable landing, while `table_merge` should stay focused on business keys and durable write policy.

Why not overwrite staging

Keeping the raw landed table intact makes debugging easier, especially when a transform or merge rule needs inspection after a failed run.

Rates use case

This node is the missing step that turns the Treasury source from a wide curve table into the long-form rows that `table_merge` actually expects.

Execution timing

Best sandwich pair`load_to_duckdb` -> `sql_transform` -> `table_merge`
