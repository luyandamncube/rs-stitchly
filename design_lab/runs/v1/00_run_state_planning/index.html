These are the top-level workflow run states we should design for. They come directly from the current runtime and API contract.

Recommended Design Order

We do not need to visually invent the lifecycle yet. We just need to agree on which states are canonical and which ones can share a family.

01 Created + Queued

These can share an early/preflight visual family, while still keeping distinct copy and state labels.

02 Planning

Planning should stand on its own because it can fail before active execution ever begins.

03 Running

This is the primary active state and will likely define most of the live observability language.

04 Succeeded

Quiet terminal success with just enough confidence and completion language.

05 Failed

Stronger error emphasis plus room for retry, diagnostics, and partial outputs.

06 Cancelling + Cancelled

These belong together as interruption states, but `cancelling` should remain visibly in progress while `cancelled` is terminal.

Node-Only States

These are real runtime states too, but they belong to node execution, not top-level workflow run surfaces.

Pending, Ready, Skipped

Useful for node rows and execution maps, but not top-level workflow run chips.

Retrying

Important for node-level observability and retry UX. We may design it later as a run-detail sub-state rather than a workflow terminal state.
