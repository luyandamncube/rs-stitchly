Node Run States

These are the canonical node-level runtime states from the current lifecycle docs and API contract. They describe what an individual node is doing inside a workflow run.

Pending pending
Known to the graph, not yet ready to execute

The node is still blocked by upstream work or graph ordering.

Design family: pre-execution
Ready ready
Eligible to execute as soon as it is scheduled

The node is now unblocked and waiting for its turn to run.

Design family: pre-execution
Running running
Actively executing right now

The node is the current live unit of work and should feel energized.

Design family: active
Succeeded succeeded
Completed successfully

The node finished and can provide quiet success confirmation.

Design family: terminal success
Failed failed
Stopped due to a node-level error

The node failed and should clearly explain where the graph broke.

Design family: terminal error
Skipped skipped
Intentionally bypassed during execution

The node did not run, but not because it failed.

Design family: de-emphasized terminal
Retrying retrying
Recovering after a failed attempt

The node is between attempts and needs special retry-aware treatment.

Design family: active nuance
Cancelling cancelling
Stopping is in progress

The node has received a stop signal, but cleanup is still active.

Design family: interruption
Cancelled cancelled
Stopped before normal completion

The node reached its final interrupted state after cancellation completed.

Recommended Design Order

We should start with the most common and visually important node states first, then layer in the more nuanced edge cases once the core node language is stable.

01 Pending + Ready

These define the pre-execution family and help us separate "not yet" from "unblocked and waiting".

02 Running

This is the most important live state and will likely drive the edge and handle language too.

03 Succeeded

Quiet confirmation of successful completion without making the node feel frozen.

04 Failed

Strong diagnostic emphasis so the graph failure point is obvious at a glance.

05 Skipped

A visibly de-emphasized state that still remains readable and intentional.

06 Retrying

Important active nuance that sits between failure and recovery, especially for integrations.

07 Cancelling + Cancelled

Interruption states are useful, but less central than the execution and retry family.
